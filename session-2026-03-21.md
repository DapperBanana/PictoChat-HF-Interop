## Bug Fixes This Session

### 1. Echo not displaying on DS (`main.rs`)

**Root cause:** `message.from` was set to the sender DS's own MAC → DS treated the echo as
a confirmation of its own outbound message, not a new received message.

**Fix:**
```rust
message.from = MACAddress::from(mac);  // mac = ESP32 host MAC: a4:f0:0f:61:9f:b0
```
DS now sees the echo as a message from the host, displays it in the chat canvas.

---

### 2. DS locked after text-only echo (`pictochat_application.rs`)

**Symptom:** Text ("hi") echoed back visually but DS could not send further messages.
Drawings (10240 bytes) worked flawlessly; text-only (2048 bytes) always locked.

**Root cause:** `EchoTransfer` handler chose `flags` based on raw frame size:
```rust
tx_out.flags = HostToClientFlags::from_bits(if echo.len() > 25 {158} else {29}).unwrap();
```

For 10240-byte drawings: last type-2 fragment = 172 bytes → `echo.len() > 25` → **flags=158** ✓
For 2048-byte text: last type-2 fragment = 16 bytes (4-byte payload) → `echo.len() ≤ 25` → **flags=29** ✗

Flags 158 (0x9E, bit 7 set) and 29 (0x1D, bit 7 clear) — bit 7 (`RESERVED_7` in `HostToClientFlags`)
appears to signal "end of data burst" to the DS. Without it on the final fragment, the DS stalls
waiting for the burst to complete.

**Fix:** Detect last type-2 fragment by checking raw bytes before choosing flags:
```rust
PictoChatState::EchoTransfer(echo) => {
    // echo[0] = type_id (u16 LE low byte) — 2 for type-2 frames
    // echo[7] = transfer_flags — 1 means last fragment (confirmed from PictochatType2 wire layout)
    let is_last_type2 = echo.len() >= 8 && echo[0] == 2 && echo[7] == 1;
    tx_out.flags = HostToClientFlags::from_bits(if is_last_type2 || echo.len() > 25 {158} else {29}).unwrap();
    tx_out.data[..echo.len()].copy_from_slice(echo.as_slice());
    tx_out.size = echo.len() as u16;
}
```

**Verified:** Text echoes back, DS can send follow-up messages. No lockups.

## PictochatType2 Wire Layout (confirmed from source)

| Byte offset | Field |
|---|---|
| 0–1 | `header.type_id` (u16 LE) |
| 2–3 | `header.size_with_header` (u16 LE) |
| 4 | `sending_console_id` (u8) |
| 5 | `payload_type` (u8) |
| 6 | `payload_length` (u8, not a struct field — inline) |
| **7** | **`transfer_flags` (u8) — 1 = last fragment** |
| 8–9 | `write_offset` (u16 LE) |
| 10–11 | `magic` ([u8; 2]) |
| 12+ | payload bytes |

## HostToClientFlags Analysis

`HostToClientFlags` is a bitflags u8 in `packets.rs`. All bits except bit 3 are named `RESERVED_*`:

| Value | Binary | Bits set | Usage |
|---|---|---|---|
| 28 (0x1C) | 00011100 | 2,3,4 | Idle/roster frames |
| 29 (0x1D) | 00011101 | 0,2,3,4 | Small/header frames |
| 30 (0x1E) | 00011110 | 1,2,3,4 | Medium frames |
| 158 (0x9E) | 10011110 | 1,2,3,4,**7** | Large frames / last fragment |

Bit 7 (0x80) is the "end of burst" signal to the DS. Any frame that terminates a data transfer
must have bit 7 set. Frame size is not a reliable proxy — use `transfer_flags == 1` instead.

## Relay Architecture Design

### Unit of relay: `MessagePayload`

The assembled `MessagePayload` in `inbound_queue` is the relay unit:

```rust
pub struct MessagePayload {
    pub magic: u8,           // 1 byte
    pub subtype: u8,         // 1 byte
    pub from: MACAddress,    // 6 bytes — set to host MAC before relay
    pub magic_1: [u8; 14],   // 14 bytes
    pub safezone: [u8; 14],  // 14 bytes
    pub message: Vec<u8>,    // 2048 or 10240 bytes (canvas bitmap)
}
// Fixed header = 36 bytes. Total = 36 + message.len()
```

`message.message` is the complete PictoChat canvas bitmap — DS-agnostic. Any DS that
joins and sends a drawing produces a `MessagePayload` with the same format.

### `from` field strategy

At the relay destination, set `message.from = host_mac` (the destination node's ESP32 MAC).
The destination DS displays the message as from the room host — acceptable for a relay where
the ESP32 represents the remote node. Sender name could be forwarded separately in Phase 4.

### TCP relay protocol (already in `main.rs`)

`server_connection_task` (lines 193–267 in `main.rs`) implements a TCP client that connects
to a relay server with simple framing:

| Field | Size | Value |
|---|---|---|
| type | 1 byte | 0x00 handshake, 0xFD data, 0xAD error |
| length | 2 bytes BE | payload length |
| payload | length bytes | `MessagePayload` serialized bytes |

Currently not spawned in `main`. For Phase 3 testing (pre-HF), spawn `server_connection_task`
and route `inbound_queue` messages to it instead of local echo.

### Relay flow (main loop changes for Phase 3)

```
Current (echo):
  inbound_queue.receive() → set from = host MAC → outbound_queue.send()

Phase 3 (relay):
  inbound_queue.receive() → serialize → TCP to relay server
  TCP from relay server → deserialize → set from = host MAC → outbound_queue.send()
```

The main loop `select` already polls both `inbound_queue` and `event_queue`. Add a third
arm for the TCP receive channel.

### "Regardless of DS" — confirmed

The handshake in `pictochat_application.rs` accepts all clients unconditionally (no MAC
filtering in `type_id == 6` handler). The assembled `MessagePayload` is the same structure
regardless of which DS sent it. No DS-specific logic needs to change for relay.
