# Session Notes — 2026-03-20

## Goal
Verify ESP32 (PictoThing) can join and leave a PictoChat session with the Nintendo DS.

## Key Architecture Clarification
**The ESP32 is the HOST, not the client.**
- PictoThing firmware sets channel 7, enables beacon TX → it acts as PictoChat Room B host
- The Nintendo DS joins as a *client* (not the other way around)
- DS must be at room selection screen (not already in a room hosting) to scan and join

### 1. GPIO34 EKV erase trigger (built + flashed)

### 2. Zero-byte allocation crash in rx_wait_loop (`pictochat_application.rs`)
Root cause: DS sends frames where `size_with_header == 0` or `data_size == 0`, causing
`vec![0u8; 0]` which the ESP32's embedded allocator rejects.

Patches in `~/foa_dswifi/foa_dswifi/src/pictochat_application.rs`:
```rust
// Guard at start of rx dispatch (after header parse):
if header.size_with_header == 0 && header.type_id != 6 {
    warn!("ignoring frame type {} with size_with_header == 0", header.type_id);
    continue;
}

// Guard in type-0 handler:
if parsed.data_size > 0 && parsed.data_size < MAX_TRANSFER_SIZE {  // was: <
    // ... alloc inflight buffer ...
}
// Also: if parsed.data_size == 0 { continue; } in realloc branch
```

## Bug Fixes Applied This Session (continued from context summary)

### 3. Double-echo causing DS to lock after first message (`main.rs`)
`EchoTransfer` in foa_dswifi already echoes frames immediately. main.rs was ALSO
echoing via `outbound_queue` → caused double-delivery → DS UI locked after message 1.
Fix: removed echo from main.rs `Either::First(message)` handler. Placeholder comment added.

### 4. Zero-byte allocation during DS handshake (`pictochat_packets.rs`)
`PictochatType2::try_from_ctx`: `payload_length: u8` could be 0 → `vec![0u8; 0]` → crash.
`MessagePayload::try_from_ctx`: `remain` could be 0 → same crash.
`MessagePayload::default()`: `vec![0; 0]` → same crash.
Fix: guard all three with `if N == 0 { Vec::new() } else { vec![0u8; N] }` / `Vec::new()`.

### 5. Comprehensive bounds guards in rx_wait_loop (`pictochat_application.rs`)
Added explicit guards for type-0 and type-2 handlers:
```rust
let _sz0 = header.size_with_header as usize;
if _sz0 == 0 || _sz0 > data_raw.len() { warn!("rx type0 bad sz={}", _sz0); continue; }

let _sz2 = header.size_with_header as usize;
if _sz2 == 0 || _sz2 > data_raw.len() { warn!("rx type2 bad sz={}", _sz2); continue; }
```
These protect against both 0-size allocations AND out-of-bounds copy_from_slice.

## Flash / Monitor Commands (updated)

**Monitor (defmt-print, works without TTY):**
```bash
# One-liner reset + monitor (on vessel-01):
python3 ~/monitor.py --reset        # resets chip, then monitors
python3 ~/monitor.py                # monitor running chip, no reset

# Or manually:
python3 -m serial.tools.miniterm /dev/ttyUSB0 115200  # raw bytes
```

**Flash (esptool, since espflash default-reset doesn't work):**
```bash
# Build:
cd ~/foa_dswifi && . ~/.cargo/env && . ~/export-esp.sh
cargo build --release -p PictoThing

# Create full 4MB image:
espflash save-image --chip esp32 --merge -T ~/foa_dswifi/table_4mb.csv \
  ~/foa_dswifi/target/xtensa-esp32-none-elf/release/PictoThing /tmp/picto_full.bin

# Flash at 0x0:
python3 -m esptool --chip esp32 --port /dev/ttyUSB0 --baud 921600 \
  write-flash --flash-size 4MB --flash-mode dio --flash-freq 40m \
  0x0 /tmp/picto_full.bin
```
Note: espflash `flash` command hangs at "Connecting..." on this board (auto-reset circuit issue).
esptool works reliably. RTS=1→LOW pulls EN low (reset), RTS=0→HIGH lets chip boot.
