## Capture Methodology

### Channel hopping
The AR9271 was placed in monitor mode via:
```bash
sudo ip link set wlan1 down
sudo iw dev wlan1 set type monitor
sudo ip link set wlan1 up
```

Channel hopping was done manually between captures via:
```bash
sudo /usr/sbin/iw dev wlan1 set channel <N>
```

### Capture command
Background capture (survived SSH disconnect) required `nohup` via:
```bash
sudo bash -c "nohup tcpdump -i wlan1 -w /tmp/ds60.pcap --immediate-mode > /tmp/tcpdump.log 2>&1 & echo PID:$!"
```
Then stop with: `sudo pkill -INT tcpdump`

### Paramiko note
`sshpass` is not installed on the Windows host. All SSH is via Python `paramiko`. Non-interactive sudo requires either `get_pty=True` + `stdin.write('SonyPlaystation1!\n')` or `sudo -S` with password piped in.

## Capture Results (6.2MB pcap, 44,539 packets)

### What we DID find

**Device: `00:25:ae:e2:fd:4a`**
- Sends 59+ probe requests throughout the 60-second window
- SSID probed: `"D86242"`
- Rate: 1.0 Mb/s, 802.11b mode, no HT (no 802.11n capabilities)
- Supported rates: 1, 2, 5.5, 11 Mb/s (+ extended 6, 9, 12, 18, 24, 36, 48, 54 Mb/s)
- DS Parameter Set IE: **channel 1** (2412 MHz) as primary
- Appears on multiple channels as we hopped (ch1, ch6, ch7, ch9, ch11) — actively scanning all channels
- **OUI `00:25:AE` is likely a Nintendo allocation** (needs verification — no local OUI DB on Pi)
- No HT capabilities IE (consistent with Nintendo handheld, not modern device)
- Sends bursts of 3–5 frames then pauses ~5 seconds (consistent with DS scanning interval)

**This device is the strongest DS candidate.** The active channel hopping and 802.11b-only bursting pattern is consistent with DS PictoChat room scanning behavior.

**Device: `34:d2:70:7b:30:4a`**
- Sends probe requests with garbled SSID `"         0pkj"` (9 spaces + `0pkj`)
- Has HT Capabilities IE — this is an **802.11n device, NOT the DS Lite**
- Ignore for DS purposes

### What we did NOT find
- No standard 802.11 beacon frames with Nintendo OUI as BSSID
- No IBSS (ad-hoc) beacons on any channel
- No Nintendo multicast frames (`03:09:bf:00:00:00`)
- No data frames from `00:25:ae:e2:fd:4a`

## Key Open Questions

1. **Do you recognise `D86242` as a WiFi network name?** (Old router, neighbor's network?) If yes, then the probe requests from `00:25:ae` may just be the DS doing background Nintendo WiFi Connection scans for a saved network — separate from PictoChat traffic.

2. **Do you have any other Nintendo devices nearby** (3DS, DSi, Wii, Switch)? OUI `00:25:AE` is more consistent with 3DS-era hardware than DS Lite. If you have a 3DS/DSi in the same room, it could be the source.

3. **Why is there no DS beacon?** Three possible explanations:
   - The DS PictoChat beacon uses a **non-standard 802.11 frame type** that the `ath9k_htc` driver doesn't relay to userspace in monitor mode. This is the most likely explanation based on DS reverse engineering literature.
   - The DS only starts beaconing after it detects another DS (room-join handshake).
   - The beacon IS being sent but on a channel we weren't on at the right moment.
